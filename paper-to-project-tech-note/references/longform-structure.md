# Longform Structure

Use this reference before outlining and during expansion.

## Goal

Write a long-form teaching note that is rich in substance, not rich in repetition.

A deep version is mandatory when the user asks for a detailed or deep explanation. Do not stop at a short overview.

## Default Structure

1. Project overview
2. Background and problem definition
3. Why the problem is hard
4. Prerequisite knowledge
5. Overall solution overview
6. Core technology 1
7. Core technology 2
8. Core technology 3
9. Core technology 4
10. Optional core technology 5-6
11. How the modules connect into a full system
12. Training and optimization logic
13. Inference or application flow
14. Result interpretation
15. Value, innovation, limits, and scope
16. Interview and presentation package
17. Appendix: glossary, formula notes, method comparisons, FAQ

## Core Technology Expansion Slots

Each core technology section must cover:

1. what it is
2. why it is needed
3. where it sits in the pipeline
4. input
5. output
6. how it works
7. what concrete problem it solves
8. difference from common alternatives
9. strengths, costs, and limitations
10. one concrete example
11. one beginner-friendly analogy
12. a short recap of its value to the full system

## Extra Depth For Key Modules

If the module is central to the method, also add:

- prerequisite knowledge for that module
- why a simpler approach is not enough
- common misunderstanding points
- how it changes the system behavior or metrics
- how it interacts with the previous and next modules

## Mandatory Thickening Pass

After drafting the full note, review it section by section.

If any section still reads like a short summary, thicken it by adding:

- richer examples
- more explicit flow breakdowns
- comparisons with alternatives
- common pitfalls
- scenario-based explanations
- links to other modules in the system
- interpretation of likely reader confusion

Do not treat this pass as optional.

## Expansion Principle

Prioritize information density over hard word counts.

Never repeat the same idea just to make the document longer.

## Quality Bar

A strong document should:

- teach enough background for a beginner
- let the reader explain the project goal in their own words
- let the reader describe 4-6 core modules clearly
- make the full pipeline understandable
- remain consistent with the interview package at the end
- feel like a deep teaching handout rather than a summary note

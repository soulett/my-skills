# PRD Writer

Generate a concise, actionable Product Requirements Document (PRD) from a rough product idea.

## What this skill does

This skill helps you produce a lightweight PRD that is easy for engineers, designers, and stakeholders to align on. It is best when a user has a vague feature or product idea and wants to quickly turn it into a scoped document with goals, assumptions, requirements, and open questions.

## When to use

Use this skill when the user wants to:
- turn an idea into a PRD
- clarify scope before implementation
- capture requirements, constraints, and goals
- document assumptions and open questions
- align engineering/design/stakeholders around a feature

Do not use this skill when the user already provided a finished PRD and only wants editing or formatting help.

## Workflow

1. Understand the product idea.
2. Ask a few clarifying questions if the scope is unclear.
3. Infer reasonable defaults when the user prefers speed over back-and-forth.
4. Produce a PRD with clear sections and practical wording.
5. Call out open issues, dependencies, and risks.

## Suggested PRD structure

Use a structure like:
- Overview
- Problem
- Goals
- Non-goals
- Users / audience
- User stories or jobs to be done
- Requirements
- Constraints
- UX / design notes
- Metrics / success criteria
- Risks / open questions
- Rollout notes

## Style guidelines

- Keep it crisp and decision-oriented.
- Prefer bullets and short paragraphs.
- Separate facts, assumptions, and open questions.
- Avoid fake precision when information is missing.
- Highlight tradeoffs explicitly.

## Example prompt patterns

- "Turn this idea into a PRD"
- "Write a PRD for a new onboarding flow"
- "Help me scope this feature before engineering starts"
- "Draft requirements for this product concept"
